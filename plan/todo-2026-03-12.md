# memolite 技术优化执行方案

> 日期：2026-03-12
> 范围：中文分词、存储体积优化
> 原则：不破坏现有功能，每项独立可回滚

---

## 背景

| 存储引擎 | 体积膨胀来源 |
|----------|------------|
| `memolite.sqlite3` | `semantic_feature_vectors` 用 JSON 序列化 64-dim float 向量，每条 ~1.17 KB；每个 episode 产生 3-50 个 derivative chunk，均需写入向量 |
| KuzuDB 目录 | `Episode`/`Derivative` 节点完整冗余存储 `content`/`content_type`/`metadata_json` 字段，而代码读取 Kuzu 时 **只使用 `uid`/`episode_uid`**，内容字段从未读取 |
| 中文分词 | `jieba` 已在代码中集成但未声明为正式依赖；未安装时降级为字符级 token，影响语义召回质量 |

## TODO-2：向量从 JSON 文本改为 Binary BLOB 存储

**预期效果**：每条 64-dim embedding 从 ~1,170 bytes（JSON TEXT）降至 256 bytes（float32 BLOB），**减少 ~78%**。所有语义搜索结果、相关度分数不变。

**风险**：低。纯序列化格式变更，不影响余弦相似度计算逻辑。需要对存量数据做一次迁移。

### 关键文件

`src/memolite/storage/sqlite_vec.py`

### 步骤

**2.1 修改 DDL（新建表时使用 BLOB）**

当前（line ~66）：
```sql
CREATE TABLE IF NOT EXISTS {table} (
    feature_id INTEGER PRIMARY KEY,
    embedding_json TEXT NOT NULL
)
```
改为：
```sql
CREATE TABLE IF NOT EXISTS {table} (
    feature_id INTEGER PRIMARY KEY,
    embedding BLOB NOT NULL
)
```

**2.2 修改写入序列化**

当前 `batch_upsert`（line ~94）：
```python
"embedding_json": json.dumps(embedding)
```
改为：
```python
"embedding": struct.pack(f"{len(embedding)}f", *embedding)
```
（`import struct` 已是标准库，无需新增依赖）

**2.3 修改读取反序列化**

当前 `search_top_k`（line ~146）：
```python
embedding = json.loads(embedding_json)
```
改为：
```python
n = len(embedding_bytes) // 4
embedding = list(struct.unpack(f"{n}f", embedding_bytes))
```

同时将 SQL 中的 `embedding_json` 字段名改为 `embedding`。

**2.4 存量数据迁移（启动时自动执行）**

在 `sqlite_vec.py` 的 `__init__` 或初始化方法中加入一次性迁移：

```python
async def _migrate_json_to_blob(self) -> None:
    """One-time migration: convert legacy JSON TEXT embeddings to BLOB."""
    async with self._conn() as conn:
        # 检查是否存在旧列
        cur = await conn.execute(
            f"SELECT COUNT(*) FROM pragma_table_info('{self._table_name}') WHERE name='embedding_json'"
        )
        row = await cur.fetchone()
        if row[0] == 0:
            return  # 已迁移或新库，跳过

        # 读出所有 JSON 向量，改写为 BLOB，重命名列
        rows = await (await conn.execute(
            f"SELECT feature_id, embedding_json FROM {self._table_name}"
        )).fetchall()

        await conn.execute(
            f"ALTER TABLE {self._table_name} ADD COLUMN embedding BLOB"
        )
        for fid, json_str in rows:
            vec = json.loads(json_str)
            blob = struct.pack(f"{len(vec)}f", *vec)
            await conn.execute(
                f"UPDATE {self._table_name} SET embedding=? WHERE feature_id=?",
                (blob, fid),
            )

        # SQLite 不支持 DROP COLUMN（3.35.0 以下），用重建表替代
        await conn.execute(
            f"CREATE TABLE {self._table_name}_new (feature_id INTEGER PRIMARY KEY, embedding BLOB NOT NULL)"
        )
        await conn.execute(
            f"INSERT INTO {self._table_name}_new SELECT feature_id, embedding FROM {self._table_name}"
        )
        await conn.execute(f"DROP TABLE {self._table_name}")
        await conn.execute(
            f"ALTER TABLE {self._table_name}_new RENAME TO {self._table_name}"
        )
        await conn.commit()
```

在初始化时调用 `await self._migrate_json_to_blob()`，仅执行一次。

**2.5 验证**


## TODO-3：KuzuDB 节点停止冗余存储内容字段

**预期效果**：KuzuDB 文件体积减少 **70-85%**。所有现有功能不受影响（已确认：代码读取 Kuzu 时只使用 `uid`/`episode_uid`，从不读取 `content`/`content_type`/`metadata_json`）。

**风险**：低-中。Kuzu 不支持 DROP COLUMN，需要分步处理存量数据库。

### 关键文件

- `src/memolite/storage/kuzu_engine.py`（schema 定义）
- `src/memolite/episodic/derivative_pipeline.py`（写入节点处）
- `src/memolite/storage/graph_store.py`（MERGE 查询处）

### 步骤

**3.1 修改 Episode 节点 schema（新库）**

文件：`src/memolite/storage/kuzu_engine.py`

当前（lines ~14-28）：
```python
CREATE NODE TABLE Episode (
    uid STRING PRIMARY KEY,
    session_id STRING,
    content STRING,          # ← 删除
    content_type STRING,     # ← 删除
    created_at STRING,
    metadata_json STRING     # ← 删除
)
```
改为：
```python
CREATE NODE TABLE Episode (
    uid STRING PRIMARY KEY,
    session_id STRING,
    created_at STRING
)
```

**3.2 修改 Derivative 节点 schema（新库）**

当前 Derivative 节点包含：`uid`, `episode_uid`, `session_id`, `content`, `content_type`, `sequence_num`, `metadata_json`

改为只保留：`uid`, `episode_uid`, `session_id`, `sequence_num`

**3.3 修改 derivative_pipeline.py 写入逻辑**

停止传递 `content`/`content_type`/`metadata_json` 到 Kuzu 节点创建。

**3.4 存量数据库迁移策略**

Kuzu 不支持 `ALTER TABLE DROP COLUMN`，提供两个选项：

**选项 A（推荐，简单）**：在 kuzu_engine.py 初始化时检测旧 schema，若存在旧字段，自动重建图：
```python
async def _migrate_kuzu_schema(self) -> None:
    """Rebuild Kuzu graph if legacy schema detected (removes redundant content fields)."""
    # 1. 检查 Episode 表是否还有 content 列
    # 2. 若有：导出所有 (uid, session_id, created_at) 和边数据
    # 3. DROP TABLE Episode, Derivative, DERIVED_FROM
    # 4. 用新 schema 重建表
    # 5. 从 SQLite episode_store 重新填充节点（无需内容字段）
    # 6. 重建 DERIVED_FROM 边（从 derivative_pipeline 记录中恢复）
```

**选项 B（保守）**：保留旧 schema 不变，仅在写入时传入空字符串 `""` 替代真实 content：
```python
# derivative_pipeline.py 中，创建节点时：
node_props = {
    "uid": episode.uid,
    "session_id": episode.session_id,
    "content": "",           # 不再存储真实内容
    "content_type": "",
    "metadata_json": "",
}
```
这样旧 schema 无需变更，体积减少效果较小（约 30-40%），但零迁移风险。

> **建议先实施选项 B**，验证无功能回退后，再实施选项 A 的 schema 精简。


## TODO-4：定期 SQLite VACUUM 清理

**预期效果**：回收软删除记录和迁移遗留的碎片空间，通常可额外减少 10-30% 文件体积。

**风险**：极低。VACUUM 是只读安全操作，不影响数据。

### 步骤

文件：`src/memolite/app/background.py`

在 `BackgroundTaskRunner` 中增加低频维护任务：

```python
_VACUUM_INTERVAL_PASSES = 500  # 每 500 次 compensation pass 执行一次

async def run_maintenance_pass(self, pass_count: int) -> None:
    """Periodic SQLite VACUUM to reclaim space from deleted records."""
    if pass_count % _VACUUM_INTERVAL_PASSES != 0:
        return
    async with self._db_conn() as conn:
        await conn.execute("VACUUM")
        await conn.commit()
```

在 `run_compensation_pass()` 的调用方处传入 pass 计数器并触发。

### 🐛 Bug-3: `vector_item_id` CRC32 碰撞风险
**位置：** `episodic/derivative_pipeline.py:170-171`

```python
def vector_item_id(uid: str) -> int:
    return zlib.crc32(uid.encode("utf-8")) & 0x7FFFFFFF
```

CRC32 是 32-bit hash，掩码后得到 31-bit 正整数，取值空间仅 ~21 亿。
根据生日悖论，约 **5 万条 derivative 记录后碰撞概率超过 50%**。

碰撞后果：两个不同 derivative 映射到同一 `feature_id`，向量 upsert 会静默覆盖，
导致检索命中错误的 episode，且无任何报错。

**修复建议：** 改用 `hashlib.sha256` 取前 8 字节得到 63-bit 正整数（标准库，无需新增依赖）：
```python
import hashlib
def vector_item_id(uid: str) -> int:
    digest = hashlib.sha256(uid.encode("utf-8")).digest()
    return int.from_bytes(digest[:8], "big") & 0x7FFFFFFFFFFFFFFF
```

## 🔴 P0 — Embedding 能力升级（核心瓶颈）

### 代码实证

**文件：** `src/memolite/app/resources.py:64-86`

```python
async def default_embedder(text: str) -> list[float]:
    dimensions = 64
    vector = [0.0] * dimensions
    tokens = _tokenize(text)
    if not tokens:
        return vector
    for token in tokens:
        digest = hashlib.blake2b(token.encode("utf-8"), digest_size=8).digest()
        bucket = int.from_bytes(digest[:4], "big") % dimensions  # 64个桶
        sign = 1.0 if (digest[4] & 1) == 0 else -1.0
        vector[bucket] += sign
    # L2 归一化后返回
    norm = math.sqrt(sum(value * value for value in vector))
    return [value / norm for value in vector]
```

这是 **随机哈希投影（Random Hash Projection）**，不是语义嵌入：
- 64 维空间极小（OpenAI text-embedding-3-small 是 1536 维）
- 语义相近的词/句子向量相似度接近随机
- "苹果手机" 和 "iPhone" 的余弦相似度与随机无异
- 跨语言（中英）匹配完全没有语义桥梁

所有语义检索的质量上限由 **token 字面重叠度** 决定，而非实际语义相似性。

**三处 hardcode 注入点：**
- `resources.py:133` → `DerivativePipeline(embedder=default_embedder)`
- `resources.py:139` → `EpisodicSearchService(embedder=default_embedder)`
- `resources.py:152` → `SemanticService(embedder=default_embedder)`

### 现有 EmbedderFn 接口

`EmbedderFn = Callable[[str], Awaitable[list[float]]]` 已在三处独立定义：
- `episodic/derivative_pipeline.py:15`
- `episodic/search.py:14`
- `semantic/service.py:17`

接口已经是函数签名，天然支持替换，无需修改调用方。

### 方案调研

#### 方案 A：本地轻量语义模型（推荐，无网络依赖）

| 模型 | 维度 | 多语言 | 大小 | 中文质量 | 推理速度(CPU) |
|------|------|--------|------|----------|--------------|
| `paraphrase-multilingual-MiniLM-L12-v2` | 384 | 50+ 语言 | ~118MB | ⭐⭐⭐ | ~10ms/句 |
| `intfloat/multilingual-e5-small` | 384 | 100+ 语言 | ~118MB | ⭐⭐⭐⭐ | ~8ms/句 |

1. DB 中存储 `embedder_name` + `embedding_dim` 元数据（schema 已有字段，见 Bug-2）
2. 提供 `repair_snapshot` CLI 工具，重新生成所有历史向量
3. 或：双写模式（旧向量保留 + 新向量并行写入，逐步迁移）

| Ollama + 本地 LLM | 免费，隐私保护，质量参差 | ⭐⭐⭐ |

**`instructor` 库优势：**
- 统一封装 Anthropic / OpenAI / Gemini / Ollama
- 自动处理 JSON 格式错误并重试

#### 方案 B：本地 NER 模型（高吞吐离线场景）

| 库 | 中文支持 | 准确率 | 适用 |
|----|----------|--------|------|

| `LAC` (百度) | ⭐⭐⭐⭐ | 高 | 快速中文 |

## 🟡 P3 — Short-term Memory 真实摘要能力

### 代码实证

**文件：** `src/memolite/memory/short_term_memory.py:104-120`

```python
async def _summarize_overflow(self) -> None:
    """Create a deterministic summary from evicted messages."""
    evicted: list[ShortTermMessage] = []
    while self._messages and self._current_message_len > self._message_capacity:
        message = self._messages.popleft()
        self._current_message_len -= len(message.content)
        evicted.append(message)

    if not evicted:
        return

    # ← 全部逻辑就是这三行字符串拼接
    summary_lines = [f"{message.producer_role}: {message.content}" for message in evicted]
    summary_chunk = " | ".join(summary_lines)
    self._summary = (
        f"{self._summary} || {summary_chunk}" if self._summary else summary_chunk
    )
    await self.persist_summary()
```

问题：
1. **越压越长：** `summary` 随对话轮次线性增长，`|| chunk1 || chunk2 || ...`
2. **无信息提炼：** 所有被驱逐的消息都完整保留在 summary 字符串中
3. **不可读：** `||` 分隔的平铺列表，不是自然语言摘要
4. **递归膨胀：** summary 本身超长后，没有 "summary of summaries" 机制

`get_context()` 的展示逻辑（`short_term_memory.py:146-155`）也只是 `f"Summary: {self._summary}"`，
直接把这个拼接字符串喂给 LLM，质量极差。

### 方案调研

| 方案 | 质量 | 成本 | 是否合适 |
|------|------|------|----------|
| LLM 摘要（Claude haiku） | ⭐⭐⭐⭐⭐ | 极低（haiku 最便宜） | ✅ 首选 |
| TextRank（提取式） | ⭐⭐⭐ | 免费 | 仅作为 fallback |
| BART/T5 本地模型 | ⭐⭐⭐⭐ | 免费，需 GPU | 可选 |
| 现有字符串拼接 | ⭐ | 免费 | ❌ 保留为 no-LLM 模式 |

**摘要触发策略调整：**
- 当前：字符数 > `message_capacity(4096)` 时驱逐最旧消息，字符串拼接
- 建议：达到阈值时 → 调用 LLM 生成结构化摘要 → 摘要替换原字符串 → 继续滑动窗口

**摘要内容应包含：**
- 关键事实（用户名字、偏好、决策）
- 待解决事项
- 情感/态度（可选）

### 代码实证

**影响：** 当前所有 episodic 检索的结果排序 = 向量余弦相似度排序，
没有考虑：query 与 episode 原文的语义匹配度、时效性、上下文相关性。

### 方案调研

#### Query Rewriting

| 方案 | 效果 | 实现难度 | 是否需要 LLM |
|------|------|----------|-------------|
| HyDE（生成假设文档） | ⭐⭐⭐⭐⭐ | 低 | 是 |
| Query 扩写（同义扩展） | ⭐⭐⭐⭐ | 低 | 是 |
| Step-back prompting | ⭐⭐⭐⭐ | 中 | 是 |
| BM25 关键词提取 | ⭐⭐⭐ | 低 | 否 |

**HyDE 原理：** 让 LLM 生成一段"假设的答案文本"，用答案的 embedding 做检索，
而不是用原始 query 的 embedding。适合记忆检索场景（"用户可能说过什么" vs "用户问什么"）。

#### Query Splitting

复合查询如 "我喜欢什么食物以及上次去的城市" 应拆分为：
- "我喜欢什么食物"
- "我上次去的城市"

分别检索后 RRF 合并结果。

#### Reranker

| 方案 | 效果 | 成本 | 是否需要额外服务 |
|------|------|------|------------------|
| Cohere Rerank API | ⭐⭐⭐⭐⭐ | 低（$1/1000次） | 需要 API key |
| cross-encoder 本地 | ⭐⭐⭐⭐ | 免费，需下载模型 | 否 |
| LLM rerank（listwise） | ⭐⭐⭐⭐ | 中 | 需要 LLM |
| BM25 混合（RRF） | ⭐⭐⭐ | 免费 | 否（无模型） |

**推荐：** 先接 Cohere Rerank（实现简单，质量高）；
本地部署需求时用 `cross-encoder/ms-marco-MiniLM-L-6-v2`（sentence-transformers 子包）。

## 🟢 P7 — 异步摄入链路增强

### 代码实证

**文件：** `src/memolite/app/background.py:77-94`

```python
async def run_compensation_pass(self) -> int:
    pending_set_ids = await self.resources.semantic_feature_store.get_history_set_ids()
    worker = SemanticIngestionWorker(...)
    processed = 0
    for set_id in pending_set_ids:          # ← 串行处理，无并发
        processed += await worker.process_pending(set_id)  # ← 一个失败会阻断后续
    ...
    self._pass_count += 1
    if self._pass_count % 500 == 0:         # ← VACUUM 频率固定，不感知实际碎片率
        await self._run_vacuum()
    return processed
```

存在问题：
1. `for set_id in pending_set_ids` → 串行，无并发，N 个 set 的延迟累加
2. 无 try/except 隔离，一个 set 失败会 raise 并阻断后续所有 set 的处理
3. VACUUM 每 500 次固定触发，不根据实际 fragmentation 动态调整
4. 无 DLQ（Dead Letter Queue），失败的 set_id 会在下次 pass 重试但没有退避
5. 无 backpressure：无论 backlog 多大，处理速度恒定

### 方案调研

| 方案 | 适用场景 | 实现复杂度 | 是否引入依赖 |
|------|----------|------------|-------------|
| asyncio.gather 并发 | 当前规模，无外部依赖 | 低 | 否 |
| asyncio.Semaphore 限流 | 防止并发过高打满 DB | 低 | 否 |
| asyncio.Queue + worker pool | 单机异步队列 | 中 | 否 |
| APScheduler | 定时任务调度 | 低 | 是（轻量） |
| Celery | 分布式生产 | 高 | 是（重） |

**推荐：** 短期用 `asyncio.gather + Semaphore + 错误隔离`，无需引入新依赖。

**动态 VACUUM 策略：**
```python
# SQLite 提供 freelist_count / page_count 可计算碎片率
result = await conn.execute("PRAGMA page_count")
result2 = await conn.execute("PRAGMA freelist_count")
fragmentation = freelist_count / page_count
if fragmentation > 0.1:  # 碎片率 > 10% 才触发 VACUUM
    await conn.execute("VACUUM")
```

## 关键依赖与风险

| 风险 | 影响 | 缓解策略 |
|------|------|----------|
| P0 升级后历史向量（64-dim）与新模型不兼容 | 高 | 提供 `repair_snapshot` CLI 一键重建；升级为可选（配置驱动） |
| LLM 抽取成本不可控（P1） | 中 | 使用 claude-haiku（最便宜）+ 缓存 + batch；regex 作为 no-cost fallback |
| Bug-3 修复后历史 feature_id 映射全部失效 | 高 | 修复 + repair 工具同步发布；分版本迁移 |
| Kùzu schema 变更无 migration 工具（P5） | 中 | 每次 schema 变更提供完整 rebuild 脚本；版本号写入 schema 元数据 |
| sentence-transformers 首次下载模型需要网络 | 低 | 支持离线预下载路径配置；hash embedder 永远作为 no-download fallback |
| 向量维度变化导致 SqliteVecIndex 存储格式变化 | 高 | 维度变化时自动触发 repair；或按维度建不同表名 |
